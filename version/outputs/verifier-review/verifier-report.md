### 1.2 缺陷严重程度枚举（Severity）❌ 不一致

| 枚举名 | 整数值(Worker3 DB) | 字符串(Worker4 API) | 一致性 |
|--------|-------------------|---------------------|--------|
| CRITICAL | 1=严重 | CRITICAL | ⚠️ 仅字符串，无整数值 |
| HIGH | 2=高 | HIGH | ⚠️ 仅字符串，无整数值 |
| MEDIUM | 3=中 | MEDIUM | ⚠️ 仅字符串，无整数值 |
| LOW | 4=低 | LOW | ⚠️ 仅字符串，无整数值 |

**问题：** Worker3数据库层明确定义了整数值1/2/3/4，但Worker4 API层仅使用字符串枚举，缺少明确的整数值映射文档。代码实现时可能产生二义性。

**推荐修正：** Worker4 API章节补充以下映射表：
```json
// severity 字段整数值与字符串映射
1 = "CRITICAL"  // 严重
2 = "HIGH"      // 高
3 = "MEDIUM"    // 中
4 = "LOW"       // 低
```

### 1.3 任务状态枚举（Task Status）❌ 严重不一致

| 枚举名 | 整数值(Worker3 DB) | 字符串(Worker4 API) | 一致性 |
|--------|-------------------|---------------------|--------|
| CREATED/PENDING | 0 | PENDING | ⚠️ 名称不匹配，无映射 |
| SCANNING/IN_PROGRESS | 1 | IN_PROGRESS | ⚠️ 名称不匹配，无映射 |
| COMPLETED | 2 | COMPLETED | ⚠️ 仅字符串，无整数值 |
| FAILED | 3 | （缺失） | ❌ API层缺少失败状态 |

**问题：**
1. Worker3: `0=创建, 1=扫描中, 2=完成, 3=失败`
2. Worker4: `PENDING, IN_PROGRESS, COMPLETED`（无FAILED）
3. 整数值和字符串之间没有明确映射
4. Worker4权限矩阵和状态流转表中完全缺失"失败"状态

**推荐修正：**
```json
// task status 字段整数值与字符串映射
0 = "PENDING"       // 已创建，等待处理
1 = "IN_PROGRESS"   // 扫描中
2 = "COMPLETED"     // 完成
3 = "FAILED"        // 失败
```

Worker4 第5.2.2节补充 `3 = FAILED` 状态，并更新状态流转表。

| CLOSED | 3 | CLOSED | ⚠️ 仅字符串，无整数值 |

```json
// issue status 字段整数值与字符串映射
0 = "OPEN"       // 未处理/开放
1 = "ASSIGNED"   // 已分配
2 = "RESOLVED"   // 已修复
3 = "CLOSED"     // 已关闭
```

## 二、问题清单

### 问题 #1：任务状态枚举严重不一致（高优先级）

**位置：** Worker3（4.3 review_task表）vs Worker4（5.2.2节）

**详情：**
- Worker3定义: `status TINYINT DEFAULT 0 COMMENT '状态:0=创建,1=扫描中,2=完成,3=失败'`
- Worker4定义: 仅 `PENDING / IN_PROGRESS / COMPLETED`，缺少 `FAILED`
- 两层之间无整数值-字符串映射文档

**影响：** 数据库存储使用整数值，API传输使用字符串，实现层必须硬编码转换逻辑，容易出错。

**修正方案：** 见上表"1.3 任务状态枚举"的推荐修正。

---

### 问题 #2：缺陷状态枚举不一致（中优先级）

**位置：** Worker3（4.4 issue表）vs Worker4（5.2.3节）

**详情：**
- Worker3定义: `0=未处理, 1=已分配, 2=已修复, 3=已关闭`
- Worker4定义: `OPEN, ASSIGNED, RESOLVED, CLOSED`
- `UNASSIGNED` 和 `OPEN` 语义略有差异

**修正方案：** 见上表"1.4 缺陷状态枚举"的推荐修正。

---

### 问题 #3：缺陷严重程度枚举缺少数值映射（中优先级）

**位置：** Worker3（4.4 issue表、4.5 rule表）vs Worker4（5.2.3节查询参数）

**详情：**
- Worker3完整定义了 `1=严重, 2=高, 3=中, 4=低`
- Worker4仅使用字符串枚举，查询参数示例: `severity: "LOW / MEDIUM / HIGH / CRITICAL"`
- 无整数值映射可能导致前端/后端通信歧义

**修正方案：** 见上表"1.2 缺陷严重程度枚举"的推荐修正。

---

### 问题 #4：错误码重叠冲突（高优先级）

**位置：** Worker4（5.1.3节）vs Worker5（7.1.1节）

**详情：**

| 异常类型 | Worker5错误码 | Worker4错误码区间 | 冲突 |
|---------|--------------|------------------|------|
| UnauthorizedException | 1000 | 1000-1999认证授权 | ❌ **重叠** |
| ForbiddenException | 1002 | 1000-1999认证授权 | ⚠️ 语义接近 |
| ResourceNotFoundException | 1003 | 9000-9999系统类 | ⚠️ 分类不一致 |
| CircuitBreakerException | 9001 | 9000-9999系统类 | ✅ 无重叠 |
| BaseException范围 | 1001-1999 | — | ⚠️ 包含1000 |

**问题：** Worker5定义了8种异常类，但错误码体系与Worker4的业务错误码完全独立。实现时可能出现同一错误码代表不同含义的情况。

**修正方案：**
统一错误码体系，推荐以Worker4的4位数字分段设计为准：
```
1000     = 未认证（UnauthorizedException）
1001-1999 = 业务校验失败
9001     = 服务熔断中（CircuitBreakerException）
9002     = 外部服务异常（ExternalServiceException）
```

Worker5的异常类应继承Worker4的错误码体系，或明确声明为内部实现细节不对外暴露。

---

### 问题 #5：架构分层描述不一致（低优先级）

**位置：** Worker1（2.1.1节）vs Worker2（3.1.1节）

**详情：**
- Worker1定义了5层: 前端展示层、BFF适配层、业务服务层、审查引擎层、数据层
- Worker2定义了4层: 前端展示层、业务逻辑层、数据访问层（缺少BFF和审查引擎的独立描述）
- Worker2的模块依赖图与Worker1的层次结构不完全对齐

**影响：** 对系统架构的整体理解可能产生歧义，但不影响实现。

**修正方案：** Worker2补充BFF适配层和审查引擎层的模块归属说明。
