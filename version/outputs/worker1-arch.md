# 代码审查系统概要设计说明书

---

## 第1章 编写目的

### 1.1 文档目的

本文档旨在全面描述代码审查系统的设计架构、功能范围和技术方案，为开发团队提供明确的技术指导，为项目评审提供完整的交付依据。本文档的主要读者包括：项目经理、开发人员、测试人员、运维人员及技术评审委员会成员。

### 1.2 项目背景

在软件工程教育与团队开发实践中，代码质量是影响项目成功的重要因素。传统的人工代码审查（Code Review）存在以下痛点：

- **效率瓶颈**：人工审查耗时较长，尤其在代码量较大的项目中，审查时间成本居高不下
- **标准不一致**：不同审查人员的标准参差不齐，难以保证审查质量的一致性
- **安全漏洞遗漏**：人工难以系统性地检测所有类型的安全漏洞
- **反馈周期长**：开发者提交代码后需等待人工审查，延长了开发迭代周期

本系统旨在构建一个自动化的代码审查平台，通过静态分析技术对代码进行系统性检测，帮助学校团队和开发团队及时发现代码质量问题、安全漏洞和编码规范违背，提升代码质量的同时大幅缩短审查周期。

### 1.2 项目目标

本系统的主要目标包括：

| 目标类别 | 具体描述 |
|---------|---------|
| **自动化代码质量检测** | 对代码进行静态分析，自动检测代码风格问题、潜在bug、复杂度超标等质量问题 |
| **安全漏洞发现** | 通过污点分析、模式匹配等技术，检测SQL注入、XSS、敏感信息泄露等安全漏洞 |
| **多语言支持** | 支持主流编程语言的代码审查，覆盖学校教学和团队开发常用的语言环境 |
| **灵活的规则配置** | 支持按项目、团队自定义审查规则，适应不同的编码规范要求 |
| **可扩展的架构** | 采用插件化设计，支持新增语言支持和新类型规则 |

### 1.3 项目范围

#### 1.3.1 支持的语言与框架

| 类别 | 支持的语言/框架 |
|------|----------------|
| **后端语言** | Java、Python、JavaScript/TypeScript、Go、C/C++ |
| **前端技术** | HTML、CSS、Vue.js、React |
| **数据格式** | JSON、YAML、XML（配置文件分析） |
| **数据库** | SQL（查询语句分析） |

#### 1.3.2 技术栈边界

| 边界类型 | 说明 |
|---------|------|
| **支持的代码规模** | 单次提交代码量不超过50,000行 |
| **支持的仓库类型** | Git（本地仓库路径） |
| **并发处理** | 支持多项目并行审查，单实例并发处理不超过5个项目 |
| **输出格式** | 支持HTML报告、JSON结果导出 |

#### 1.3.3 非功能边界

- **运行环境**：支持Docker容器化部署
- **数据存储**：支持MySQL持久化存储，Redis缓存加速
- **不包含范围**：不包含代码动态运行测试、不包含依赖漏洞扫描（待后续版本扩展）

### 1.4 术语表

| 术语 | 英文全称 | 定义说明 |
|------|---------|---------|
| **AST** | Abstract Syntax Tree | 抽象语法树，将源代码转换为树形结构表示，便于程序分析 |
| **污点分析** | Taint Analysis | 一种静态分析技术，跟踪不可信数据（污点）的传播路径，检测数据是否被不当使用 |
| **RBAC** | Role-Based Access Control | 基于角色的访问控制，通过角色分配权限来管理用户对系统资源的访问 |
| **BFF** | Backend For Frontend | 前后端适配层，为前端提供定制化的API服务 |
| **规则引擎** | Rule Engine | 用于定义、执行和管理业务规则的核心组件 |
| **LGTM** | Looks Good To Me | 代码审查通过的行话，表示代码审查结果良好 |
| **SAST** | Static Application Security Testing | 静态应用安全测试，无需运行程序即可发现安全漏洞 |
| **CLI** | Command Line Interface | 命令行界面，本系统的终端交互工具 |
| **WebHook** | WebHook | 事件回调机制，用于与外部系统（如Git）集成 |

## 第2章 系统架构设计

### 2.1 系统分层架构

本系统采用分层架构设计，将系统划分为五个主要层次，各层职责清晰、边界明确，便于独立开发和维护。

#### 2.1.1 架构层次概览

```
┌─────────────────────────────────────────────────────────────────┐
│                        前端展示层 (Web UI)                       │
│                    Vue.js + Element Plus 构建                    │
├─────────────────────────────────────────────────────────────────┤
│                       BFF适配层 (API Gateway)                    │
│                   Spring Cloud Gateway 提供                      │
├─────────────────────────────────────────────────────────────────┤
│                       业务服务层 (Business Service)              │
│              用户服务 / 项目服务 / 审查任务服务 / 报告服务          │
├─────────────────────────────────────────────────────────────────┤
│                      审查引擎层 (Review Engine)                   │
│     Java审查引擎   │   Python审查引擎   │   JS审查引擎           │
│     (ECharts SA)  │   (PyLint/AST)    │   (ESLint)             │
├─────────────────────────────────────────────────────────────────┤
│                        数据层 (Data Layer)                       │
│            MySQL (持久化)  │  Redis (缓存)  │  文件存储             │
└─────────────────────────────────────────────────────────────────┘
```

#### 2.1.2 各层职责说明

| 层级 | 组件名称 | 主要职责 |
|------|---------|---------|
| **前端层** | Web Console | 提供项目管理、审查配置、结果查看的用户界面 |
| **BFF层** | API Gateway | 请求路由、鉴权、协议转换、限流保护 |
| **业务服务层** | User Service | 用户认证、RBAC权限管理 |
| **业务服务层** | Project Service | 项目管理、仓库配置、规则绑定 |
| **业务服务层** | Review Task Service | 审查任务调度、状态管理、结果聚合 |
| **业务服务层** | Report Service | 生成审查报告、导出功能 |
| **审查引擎层** | Java Engine | Java代码AST解析、规则检查 |
| **审查引擎层** | Python Engine | Python代码AST解析、规则检查 |
| **审查引擎层** | JS Engine | JavaScript/TypeScript代码分析 |
| **数据层** | MySQL | 存储用户、项目、审查记录等核心数据 |
| **数据层** | Redis | 缓存热点数据、分布式锁、任务队列 |
| **数据层** | File Storage | 存储审查结果附件、报告文件 |

#### 2.1.3 层间通信约定

| 调用方向 | 通信方式 | 说明 |
|---------|---------|------|
| 前端 → BFF | RESTful HTTP | JSON格式请求响应 |
| BFF → 业务服务 | HTTP/RPC | Feign远程调用 |
| 业务服务 → 引擎层 | 内部调用 / 消息队列 | 异步任务通过RabbitMQ解耦 |
| 业务服务 → 数据层 | JDBC / Redis Client | ORM框架访问MySQL，客户端访问Redis |

### 2.4 设计原则

#### 2.4.1 核心设计原则

| 原则名称 | 描述 | 在本系统中的应用 |
|---------|------|-----------------|
| **无状态服务原则** | 服务实例不保存会话状态，所有状态外置到缓存或数据库 | 所有微服务设计为无状态，通过Redis管理会话和分布式锁 |
| **规则可配置原则** | 审查规则与引擎解耦，支持运行时配置和热更新 | 规则存储于数据库，支持通过API动态管理规则 |
| **引擎隔离原则** | 不同语言的审查引擎相互隔离，单引擎故障不影响其他 | 每种语言对应独立引擎模块，通过消息队列异步调用 |
| **面向失败设计** | 系统各组件考虑容错、熔断、降级策略 | 引入Sentinel/Hystrix实现熔断保护 |
| **可观测性原则** | 日志、指标、追踪全面覆盖 | 集成Spring Boot Actuator + Micrometer |

#### 2.4.2 无状态服务实现

```
┌─────────────────────────────────────────────────────────────────┐
│                     请求处理流程（无状态）                        │
│                                                                   │

```

#### 2.4.4 引擎隔离策略

| 隔离维度 | 实现方案 | 故障影响范围 |
|---------|---------|-------------|
| **进程隔离** | 不同语言引擎运行在独立进程中 | 单引擎崩溃不影响其他引擎 |
| **资源隔离** | Docker容器CPU/内存限制 | 单引擎OOM不影响整体服务 |
| **数据隔离** | 独立数据库Schema | 引擎间数据无交叉依赖 |
| **通信隔离** | 消息队列异步通信 | 引擎响应延迟不影响主服务 |
