## 3.2 子模块详细说明

### 3.2.1 代码审查模块（Code Review Service）

#### 3.2.1.1 功能描述

代码审查模块是系统的核心业务模块，负责接收代码提交、解析源代码、执行缺陷检测并存储检测结果。该模块实现了代码审查的完整流程，是系统价值的直接体现。

#### 3.2.1.2 核心功能

| 功能项 | 说明 |
|--------|------|
| 提交接收 | 支持Git Webhook、REST API、CLI工具三种提交方式 |
| 格式解析 | 支持zip、tar.gz压缩包及单文件上传，单次提交最大100MB |
| 多语言解析 | 支持Java/Python/JavaScript/Go，基于ANTLR生成AST |
| 缺陷检测 | 内置50+安全规则，覆盖空指针、SQL注入、XSS、硬编码密码等 |
| 结果存储 | 检测结果持久化到数据库，支持历史查询 |

#### 3.2.1.3 核心接口

```
• POST /api/v1/submissions          - 提交代码审查
• POST /api/v1/submissions/upload   - 上传代码压缩包
• GET  /api/v1/submissions/{id}     - 查询提交状态
• POST /api/v1/review/start/{id}   - 启动审查任务
• GET  /api/v1/issues/{submission_id} - 获取缺陷列表
```

#### 3.2.1.4 边界约束

- 单次提交文件大小限制：100MB
- 单个项目最大文件数：5000个
- 最大单文件行数：50000行
- 支持的文件编码：UTF-8、GBK
- 单文件解析超时：30秒

#### 3.2.1.5 输入输出

**输入**：源代码文件（压缩包或单文件）、审查配置（可选）

**输出**：缺陷列表，包含文件路径、行号、缺陷类型、严重程度、修复建议

### 3.2.2 规则管理模块（Rule Management Service）

#### 3.2.2.1 功能描述

规则管理模块负责管理和维护代码审查规则库，支持规则的注册、匹配、执行和热更新。该模块是代码审查系统的决策中心，确保审查标准的一致性和可配置性。

#### 3.2.2.2 核心功能

| 功能项 | 说明 |
|--------|------|
| 规则注册表 | 维护规则元数据，支持规则分类（安全/性能/规范）和标签管理 |
| 规则匹配器 | 根据语言类型、文件路径等条件智能匹配适用规则 |
| 规则执行器 | 按优先级执行规则，支持并行执行和结果聚合 |
| 热更新机制 | 运行时加载/卸载规则，配置变更10秒内生效 |
| 自定义规则 | 支持YAML格式自定义规则配置 |

#### 3.2.2.3 核心接口

```
• GET  /api/v1/rules                - 获取规则列表
• POST /api/v1/rules               - 注册新规则
• PUT  /api/v1/rules/{id}          - 更新规则
• DELETE /api/v1/rules/{id}        - 删除规则
• POST /api/v1/rules/reload        - 重新加载规则
• GET  /api/v1/rules/{id}/status   - 查询规则执行状态
```

#### 3.2.2.4 边界约束

- 规则配置文件格式：YAML
- 单个规则文件最大：100KB
- 规则数量上限：单个规则集最多1000条规则
- 规则执行超时：单条规则最大5秒
- 热更新生效延迟：配置变更后10秒内

#### 3.2.2.5 内置规则分类

| 类别 | 规则数量 | 示例 |
|------|----------|------|
| 安全漏洞 | 25+ | SQL注入、XSS、路径遍历、硬编码密码 |
| 空值处理 | 10+ | 空指针检查、空集合遍历 |
| 代码规范 | 15+ | 过长方法、重复代码、魔法数字 |

### 3.2.3 报告生成模块（Report Generation Service）

#### 3.2.3.1 功能描述

报告生成模块负责将缺陷检测结果转换为可读的审查报告，支持多种输出格式，并可导出到飞书文档进行协作分享。

#### 3.2.3.2 核心功能

| 功能项 | 说明 |
|--------|------|
| 多格式导出 | 支持HTML、PDF、JSON三种格式 |
| 多维度分组 | 按文件、按缺陷类型、按提交人分组展示 |
| 飞书集成 | 支持将报告导出为飞书云文档 |
| 差异化视图 | 提供概览视图、详情视图、趋势视图 |
| 报告模板 | 支持自定义报告模板 |

#### 3.2.3.3 核心接口

```
• POST /api/v1/reports              - 生成审查报告
• GET  /api/v1/reports/{id}         - 获取报告详情
• GET  /api/v1/reports/{id}/download - 下载报告文件
• POST /api/v1/reports/export/feishu - 导出到飞书文档
• GET  /api/v1/reports/templates    - 获取报告模板列表
```

#### 3.2.3.4 边界约束

- HTML报告最大大小：50MB
- PDF报告最大页数：500页
- JSON报告最大记录数：100000条
- 飞书文档标题长度：不超过100字符
- 报告生成超时：大型项目报告生成不超过60秒

#### 3.2.3.5 报告内容结构

```
报告内容
├── 审查概览
│   ├── 提交信息（提交人、时间、项目名）
│   ├── 总体统计（文件数、缺陷总数、严重程度分布）
│   └── 审查结论（通过/需修复）
├── 缺陷详情
│   ├── 按文件分组
│   ├── 按类型分组
│   └── 按严重程度分组
└── 附录
    ├── 规则说明
    └── 修复建议
```

### 3.2.4 统计分析模块（Statistics Analysis Service）

#### 3.2.4.1 功能描述

统计分析模块负责对代码审查数据进行统计和分析，提供多维度的审查趋势、质量评估和问题分布分析，帮助教师和班级管理员了解代码质量状况。

#### 3.2.4.2 核心功能

| 功能项 | 说明 |
|--------|------|
| 审查统计 | 统计审查次数、缺陷数量、修复率等关键指标 |
| 趋势分析 | 按日/周/月展示缺陷数量和类型的变化趋势 |
| 问题分布 | 分析缺陷在不同文件、类型、严重程度的分布 |
| 质量评估 | 对学生代码质量进行评分和排名 |
| 报表导出 | 支持导出统计报表为Excel格式 |

#### 3.2.4.3 核心接口

```
• GET  /api/v1/stats/overview       - 获取统计概览
• GET  /api/v1/stats/trend          - 获取趋势数据
• GET  /api/v1/stats/distribution   - 获取问题分布
• GET  /api/v1/stats/ranking       - 获取质量排名
• GET  /api/v1/stats/export         - 导出统计报表
```

#### 3.2.4.4 边界约束

- 统计时间范围：支持自定义，最大1年
- 排名数据范围：支持班级内排名、全校排名
- 报表导出超时：不超过30秒
- 缓存策略：统计数据缓存30分钟

#### 3.2.4.5 统计指标体系

| 指标类别 | 具体指标 |
|----------|----------|
| 审查规模 | 审查次数、提交人数、文件总数、代码行数 |
| 缺陷情况 | 缺陷总数、严重缺陷数、高危缺陷数、缺陷密度 |
| 修复情况 | 修复率、平均修复时间、重复提交率 |
| 质量评分 | 代码质量分、排名变化、改进幅度 |

### 3.2.5 班级教师管理模块（Class Management Service）

#### 3.2.5.1 功能描述

班级教师管理模块负责管理班级、教师、学生三层组织架构，支持课程配置、学生管理、教师权限管理等教学管理功能，是系统面向教育场景的核心模块。

#### 3.2.5.2 核心功能

| 功能项 | 说明 |
|--------|------|
| 班级管理 | 创建/编辑/删除班级，配置班级审查规则 |
| 教师管理 | 添加/移除教师，分配班级管理权限 |
| 学生管理 | 添加/移除学生，批量导入学生账号 |
| 课程配置 | 配置课程审查规则、评分标准 |
| 数据导入 | 支持CSV批量导入学生信息 |
| 通知功能 | 向学生/教师发送审查结果通知 |

#### 3.2.5.3 核心接口

```
# 班级管理
• POST /api/v1/classes              - 创建班级
• GET  /api/v1/classes              - 获取班级列表
• PUT  /api/v1/classes/{id}         - 更新班级
• DELETE /api/v1/classes/{id}      - 删除班级

# 教师管理
• POST /api/v1/teachers             - 添加教师
• GET  /api/v1/teachers             - 获取教师列表
• DELETE /api/v1/teachers/{id}      - 移除教师

# 学生管理
• POST /api/v1/students             - 添加学生
• POST /api/v1/students/batch       - 批量导入学生
• GET  /api/v1/students             - 获取学生列表
• DELETE /api/v1/students/{id}      - 移除学生
```

#### 3.2.5.4 边界约束

- 单个班级最大学生数：200人
- 单个教师最大管理班级数：10个
- CSV导入最大行数：500行
- 数据隔离级别：班级（Class）级别
- 操作权限：仅班级管理员和超级管理员可操作

#### 3.2.5.5 数据模型

```
班级 (Class)
├── 班级ID
├── 班级名称
├── 所属学校/院系
├── 创建时间
└── 配置信息（审查规则、评分标准）

教师 (Teacher)
├── 教师ID
├── 姓名
├── 关联班级列表
├── 角色权限
└── 创建时间

学生 (Student)
├── 学生ID
├── 姓名
├── 学号
├── 关联班级
├── 提交统计
└── 质量评分
```

### 3.2.6 调度管理模块（Scheduling Management Service）

#### 3.2.6.1 功能描述

调度管理模块负责管理代码审查任务的调度和执行，支持任务队列管理、并发控制、超时重试等机制，确保审查任务高效、稳定地执行。该模块是系统任务执行的核心调度中枢。

#### 3.2.6.2 核心功能

| 功能项 | 说明 |
|--------|------|
| 任务队列 | 基于Redis实现分布式任务队列，支持任务优先级 |
| 并发控制 | 限制同时执行的扫描任务数量，避免资源竞争 |
| 超时重试 | 支持任务执行超时检测和自动重试 |
| 任务状态追踪 | 实时跟踪任务执行状态，支持取消和暂停 |
| AST解析异常兜底 | 单文件解析异常时跳过并记录，继续处理其他文件 |

#### 3.2.6.3 AST解析异常兜底策略

为保证任务调度的鲁棒性，调度管理模块实现了完善的AST解析异常兜底机制：

##### 3.2.6.3.1 异常分类处理

| 异常类型 | 处理策略 | 日志级别 | 影响范围 |
|---------|---------|---------|---------|
| 语法解析错误 | 跳过该文件，记录失败详情 | ERROR | 单文件 |
| 编码识别失败 | 尝试多编码重试，仍失败则跳过 | WARN | 单文件 |
| 解析超时 | 跳过该文件，超时计数器+1 | WARN | 单文件 |
| 内存溢出风险 | 跳过大文件，触发告警 | WARN | 单文件 |

##### 3.2.6.3.2 降级执行流程

```
任务调度执行流程
┌─────────────────────────────────────────────────────────────┐
│  1. 接收审查任务请求                                          │
│  2. 入队到Redis任务队列                                        │
│  3. Worker拉取任务                                            │
│  4. 分发文件列表给解析器                                       │
│  5. 逐文件执行AST解析                                         │
│     ├─ 解析成功 → 执行规则匹配                                │
│     └─ 解析失败 → 记录失败文件 + 继续处理下一文件             │
│  6. 汇总结果（部分成功 + 失败文件列表）                         │
│  7. 更新任务状态                                              │
│  8. 通知任务完成                                               │
└─────────────────────────────────────────────────────────────┘
```

##### 3.2.6.3.3 部分结果返回机制

当发生解析异常时，系统保证返回最大可用结果：

```json
{
    "taskId": "task-uuid-456",
    "status": "PARTIAL_COMPLETED",
    "summary": {
        "totalFiles": 100,
        "successfulFiles": 95,
        "failedFiles": 5,
        "successRate": "95%"
    },
    "results": {
        "issues": [...],
        "failedFileList": [
            {
                "path": "src/broken.java",
                "reason": "PARSE_ERROR",
                "line": 42,
                "detail": "unexpected token '}'"
            }
        ]
    },
    "recommendation": "建议检查并修复5个解析失败的文件后重新提交审查"
}
```

##### 3.2.6.3.4 重试策略配置

```yaml
scheduler:
  retry:
    max-attempts: 3
    backoff-multiplier: 2
    initial-delay-ms: 1000
    max-delay-ms: 30000
  timeout:
    per-file-ms: 30000
    total-task-minutes: 60
  fallback:
    on-parse-error: skip-and-continue
    on-timeout: skip-and-continue
    report-failed-files: true
```

#### 3.2.6.4 核心接口

```
• POST /api/v1/tasks              - 创建审查任务
• GET  /api/v1/tasks/{id}        - 查询任务状态
• POST /api/v1/tasks/{id}/cancel - 取消任务
• GET  /api/v1/tasks/{id}/results - 获取任务结果
• GET  /api/v1/queue/stats        - 获取队列统计
```

#### 3.2.6.5 边界约束

- 单个任务最大文件数：5000个
- 单个任务最大超时时间：60分钟
- 单文件解析超时：30秒
- 最大并发任务数：10个
- 任务队列容量：1000个

## 3.4 模块设计原则

1. **单一职责原则**：每个模块只负责一个明确的业务领域
2. **接口隔离原则**：模块间通过定义良好的接口通信，避免直接依赖
3. **依赖倒置原则**：高层模块不直接依赖低层模块，通过接口抽象
4. **可扩展性原则**：支持规则热更新和插件扩展
5. **数据隔离原则**：班级级别数据隔离，确保教学数据安全
