## 3. 网关层（Spring Cloud Gateway）⭐ **本版本新增**

### 3.1 为什么独立出网关

按概要设计 2.1 要求，**所有 HTTP 请求先经网关再到业务服务**。独立网关的好处：

1. **统一鉴权**：JWT 校验、角色粗粒度拦截集中在一处，业务服务无需重复 Security 配置
2. **限流集中**：Redis + Lua 实现令牌桶/滑动窗口，全平台单点限流
3. **跨域统一**：CORS 仅在网关配置，业务服务完全无需 CORS
4. **路由分发**：URL path → serviceId，可按 service 灰度（金丝雀）
5. **熔断降级**：业务服务异常时网关直接返回兜底
6. **审计日志**：所有请求经过网关，方便统一记录 IP/UA/时间
7. **未来扩展**：加 Nacos 服务发现、加 Spring Cloud Config 配置中心，零业务代码改动

### 3.2 技术选型

| 项 | 选型 | 理由 |
|----|------|------|
| 框架 | Spring Cloud Gateway 4.x | 响应式 WebFlux、性能优于 Zuul |
| 鉴权 | 自定义 `GlobalFilter` + JWT | 不依赖业务服务，零信任 |
| 限流 | Redis + Lua 令牌桶 | 分布式原子限流 |
| 跨域 | Gateway 内置 CORS 配置 | 业务服务不配 |
| 熔断 | Resilience4j | Spring 生态标准 |
| 注册中心 | 暂用静态路由（`lb://`） | 简化部署；未来可换 Nacos |
| 服务发现 | 无（静态配置） | 单体业务，YAML 列出 |

### 3.3 路由配置

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: code-audit-server
          uri: lb://code-audit-server
          predicates:
            - Path=/api/**
          filters:
            - StripPrefix=1            # /api/auth/login -> /auth/login
            - name: RequestTime
            - name: CircuitBreaker
              args:
                name: backendCB
                fallbackUri: forward:/fallback
        - id: code-audit-doc
          uri: lb://code-audit-server
          predicates:
            - Path=/v3/api-docs/**,/doc.html,/webjars/**
```

### 3.4 全局过滤器链

执行顺序（数字越小越靠前）：

| Order | 过滤器 | 职责 |
|-------|--------|------|
| -100 | `RequestLoggingFilter` | 记录 method/path/ip/ua/耗时 |
| -50 | `CorsGatewayFilter` | 注入 CORS 响应头 |
| 0 | `JwtAuthFilter` | 解析 Authorization，提取 uid/role |
| 10 | `RoleCheckFilter` | 检查受限路径的角色 |
| 20 | `RateLimitFilter` | Redis 令牌桶限流 |
| 100 | `RetryFilter` | GET 接口失败重试 2 次（间隔 1s） |

### 3.5 鉴权流程

```
HTTP 请求 → Nginx → Gateway
                          │
                          ▼
                  JwtAuthFilter
                          │
        ┌─────────────────┴──────────────────┐
        │  解析 Authorization: Bearer xxx    │
        │  验签 + 是否在黑名单              │
        │  提取 uid / role 放入 ServerWebExchange attributes
        │  设置 X-User-Id / X-User-Role 响应头回传业务服务
        └─────────────────┬──────────────────┘
                          ▼
                  Path 白名单？(/api/auth/login ...)
                          │
            ┌─────────────┴────────────┐
            │ 白名单 → 直接转发         │
            │ 需鉴权 → 校验通过后转发   │
            │        校验失败 → 401    │
            └─────────────┬────────────┘
                          ▼
                  RateLimitFilter
                  (单用户每分钟 5 次审查)
                          │
                          ▼
                  路由至业务服务
                  (X-User-Id: 123, X-User-Role: TEACHER)
```

### 3.6 限流（Redis 令牌桶）

**Key**：`ratelimit:{userId}:{path}:{minute}`
**Value**：`{count}:{expireAt}`（INCR + EXPIRE）
**Lua 脚本**：原子 INCR + EXPIRE 第一次时设置 TTL

```lua
-- ratelimit.lua
local current = redis.call('INCR', KEYS[1])
if current == 1 then
    redis.call('EXPIRE', KEYS[1], ARGV[1])
end
if current > tonumber(ARGV[2]) then
    return 0
end
return 1
```

业务服务无需感知限流逻辑。

### 3.7 业务服务契约（上下游）

#### 3.7.1 网关 → 业务服务的请求头
| Header | 含义 | 示例 |
|--------|------|------|
| `X-User-Id` | 当前用户 ID（由网关解析 JWT 后注入） | `123` |
| `X-User-Role` | 当前用户角色 | `TEACHER` |
| `X-User-Name` | 用户名（可选） | `teacher` |
| `X-Request-Id` | 请求唯一 ID（UUID） | `a1b2c3-...` |
| `X-Real-IP` | 客户端真实 IP | `192.168.1.10` |

#### 3.7.2 业务服务的简化
由于鉴权在网关完成，业务服务：
- ✅ **可移除**：`SecurityConfig` 中对 Token 的解析（保留用户态上下文）
- ✅ **可移除**：`JwtAuthenticationFilter`（网关已处理）
- ✅ **可移除**：CORS 配置
- ✅ **保留**：业务注解 `@PreAuthorize`、方法级权限（兜底）
- ✅ **保留**：`SecurityUtil.currentUserId()` —— **从 header `X-User-Id` 读取**而非从 JWT 解析

这是本版本最大改动：`SecurityUtil` 重构为 `WebFilter` / `HandlerInterceptor`，从 request header 读取用户信息。

### 4.4 与网关的接口契约

业务服务只接收请求 + 读 `X-User-Id` / `X-User-Role` header：
```java
public static Long currentUserId() {
    ServletRequestAttributes attrs = (ServletRequestAttributes) RequestContextHolder.getRequestAttributes();
    if (attrs == null) throw new UnauthorizedException("no request");
    HttpServletRequest req = attrs.getRequest();
    String uid = req.getHeader("X-User-Id");
    if (uid == null) throw new UnauthorizedException("未登录");
    return Long.valueOf(uid);
}
```

返回结果**不**写自定义头，全部走业务字段。

## 5. 数据层（MySQL 8.0 + Redis 7.0）

### 5.1 选型
- **MySQL 8.0**：关系型，支持窗口函数、CTE；utf8mb4 支持 emoji；性能稳定
- **Redis 7.0**：内存缓存，支持发布订阅、Streams；规则缓存 + JWT 黑名单 + 限流

### 5.2 MySQL 设计

#### 5.2.1 6 张核心表

| 表 | 用途 | 关键索引 |
|----|------|---------|
| `user` | 用户 | uk_username, idx_role |
| `review` | 审查主表 | idx_user_id, idx_review_time, idx_status |
| `issue` | 审查问题 | idx_review_id, idx_severity, idx_review_severity（联合） |
| `rule` | 规则 | uk_name, uk_code, idx_category, idx_enabled, idx_class_id |
| `class_group` | 班级 | idx_teacher_id |
| `class_user` | 班级成员 | uk_class_student, idx_student_id |

详细字段、类型、约束、ER 图、迁移策略参见《数据层详细设计说明书》。

#### 5.2.2 软删除
- 所有表 `is_deleted TINYINT DEFAULT 0`
- MyBatis-Plus `@TableLogic` 自动转换 `DELETE` 为 `UPDATE is_deleted=1`
- 查询自动追加 `WHERE is_deleted=0`

#### 5.2.3 性能
- 单表 1000 万行以下无需分库分表
- 关键查询使用覆盖索引（`idx_review_severity` 覆盖 `review_id + severity`）
- 慢查询监控：MySQL slow_query_log

### 5.3 Redis 设计

#### 5.3.1 Key 设计

| Key | 类型 | 内容 | TTL | 操作方 |
|-----|------|------|-----|--------|
| `code-audit:rules:enabled` | String (JSON) | 启用的规则列表 | 永久（手动失效） | 后端读写、网关读 |
| `jwt:blacklist:{token}` | String | "1" | Token 剩余有效期 | 后端写、网关读 |
| `ratelimit:{userId}:{path}:{minute}` | String (count) | 计数 | 60s | 网关写读 |
| `session:{userId}`（V2 预留） | Hash | 用户会话 | 30min | 后端写、网关读 |

#### 5.3.2 缓存策略

**加载**：应用启动时从 MySQL 加载启用的规则到 Redis
**失效**：规则 CRUD 后 `DEL code-audit:rules:enabled` + `PUBLISH code-audit:rule:updated`
**降级**：Redis 不可用时使用进程内 `AtomicReference` 兜底

#### 5.3.3 序列化
统一使用 Jackson JSON（`Jackson2JsonRedisSerializer`）
复杂对象使用 `Hash`（如 session）

### 5.4 与业务服务接口

业务服务通过 Spring Data Redis 操作：
```java
@Resource RedisTemplate<String, Object> redisTemplate;

List<Rule> rules = (List<Rule>) redisTemplate.opsForValue().get("code-audit:rules:enabled");
```

### 5.5 备份与恢复

| 资源 | 策略 | 频率 |
|------|------|------|
| MySQL | mysqldump 全量 + binlog 增量 | 每天凌晨 2 点全量，binlog 实时 |
| Redis | RDB + AOF 混合 | 每 5 分钟 RDB，AOF 每次写 |

## 6. 接口契约（4 层之间）

### 6.1 表现层 ↔ 网关层

| 方向 | 协议 | 数据 |
|------|------|------|
| 前端 → 网关 | HTTPS / JSON | `{ method, url, headers, body }` |
| 网关 → 前端 | HTTPS / JSON | `{ code, message, data }` |

请求头约定：
- `Authorization: Bearer {token}`（除公开接口外）
- `Content-Type: application/json`
- `Accept-Language: zh-CN`

### 6.2 网关层 ↔ 业务层

| 方向 | 数据 |
|------|------|
| 网关 → 业务 | HTTP 转发 + 注入 header：`X-User-Id`、`X-User-Role`、`X-Request-Id` |
| 业务 → 网关 | HTTP 响应，原样回传 |
| 网关 → 业务 | 异常时：业务返回非 2xx，网关包装为统一格式 |

### 6.3 业务层 ↔ 数据层

| 方向 | 协议 |
|------|------|
| 业务 → MySQL | JDBC + MyBatis-Plus |
| 业务 → Redis | Lettuce + Spring Data Redis |
| MySQL → 业务 | ResultSet |
| Redis → 业务 | 反序列化对象 |

## 7. 部署拓扑（完整）

### 7.1 容器编排

```yaml
# docker-compose.yml
services:
  mysql:        # 3306
  redis:        # 6379
  backend:      # 8080（仅内网）
  gateway:      # 8080（仅内网）
  nginx:        # 80/443（对外）
```

### 7.2 启动顺序

1. `mysql` healthcheck 通过
2. `redis` healthcheck 通过
3. `backend` 启动并完成 Flyway/init.sql 初始化
4. `gateway` 启动（依赖 backend）
5. `nginx` 启动（依赖 gateway）

### 7.3 网络

所有服务在 `code-audit-net` bridge 网络内互通。
- mysql / redis **不对外暴露端口**（仅内网访问）
- backend / gateway **不对外暴露端口**（仅通过 nginx 入口）
- nginx **对外暴露 80/443**

```yaml
mysql:
  # 不写 ports，仅内网
  expose: ["3306"]
```

### 7.4 资源限制

| 容器 | CPU | 内存 | 说明 |
|------|-----|------|------|
| mysql | 1.0 | 1G | 8.0 默认即可 |
| redis | 0.5 | 256M | 规则缓存 + 限流 |
| backend | 2.0 | 1.5G | Java 堆 1G |
| gateway | 1.0 | 512M | WebFlux 更轻量 |
| nginx | 0.25 | 64M | 静态资源 |
