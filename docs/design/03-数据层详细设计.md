### 1.2 表详细设计

#### 1.2.1 user（用户表）

| 字段 | 类型 | 长度 | 约束 | 默认 | 说明 |
|------|------|------|------|------|------|
| id | BIGINT | 20 | PK, AUTO_INCREMENT | - | 主键 |
| username | VARCHAR | 50 | NOT NULL, UNIQUE | - | 用户名，登录凭证 |
| password | VARCHAR | 255 | NOT NULL | - | BCrypt 加密后约 60 字符 |
| role | VARCHAR | 20 | NOT NULL | 'STUDENT' | STUDENT / TEACHER / ADMIN |
| email | VARCHAR | 100 | NULL | NULL | 邮箱 |
| real_name | VARCHAR | 50 | NULL | NULL | 真实姓名 |
| create_time | DATETIME | - | NOT NULL | CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | - | NOT NULL | ON UPDATE CURRENT_TIMESTAMP | 更新时间 |
| is_deleted | TINYINT | 1 | NOT NULL | 0 | 逻辑删除 |

**索引**：
- `PRIMARY KEY (id)`
- `UNIQUE KEY uk_username (username)` —— 用户名唯一
- `KEY idx_role (role)` —— 按角色查询（管理员分页）
- `KEY idx_is_deleted (is_deleted)` —— 配合软删除

**预估数据量**：500-2000 行（中型高校院系规模），无性能压力

#### 1.2.2 review（审查记录表）

| 字段 | 类型 | 长度 | 约束 | 默认 | 说明 |
|------|------|------|------|------|------|
| id | BIGINT | 20 | PK, AUTO_INCREMENT | - | 主键 |
| user_id | BIGINT | 20 | NOT NULL | - | 提交用户 |
| code_content | MEDIUMTEXT | 16MB | NOT NULL | - | 原始代码（最多 500KB，MEDIUMTEXT 留余量） |
| file_name | VARCHAR | 255 | NULL | NULL | 文件名 |
| line_count | INT | - | NOT NULL | 0 | 代码行数 |
| issue_count | INT | - | NOT NULL | 0 | 问题总数（冗余字段） |
| status | VARCHAR | 20 | NOT NULL | 'PENDING' | PENDING / COMPLETED / FAILED |
| error_msg | VARCHAR | 500 | NULL | NULL | 失败原因 |
| review_time | DATETIME | - | NOT NULL | CURRENT_TIMESTAMP | 审查时间 |
| cost_ms | BIGINT | - | NULL | NULL | 耗时（毫秒） |
| is_deleted | TINYINT | 1 | NOT NULL | 0 | 软删除 |

**索引**：
- `PRIMARY KEY (id)`
- `KEY idx_user_id (user_id)` —— 个人历史查询
- `KEY idx_review_time (review_time)` —— 按时间排序
- `KEY idx_status (status)` —— 状态过滤
- `KEY idx_is_deleted (is_deleted)`

**联合查询优化**：个人历史分页
```sql
SELECT * FROM review
WHERE user_id = ? AND is_deleted = 0
ORDER BY review_time DESC
LIMIT ? OFFSET ?;
```
使用 `idx_user_id` 即可，ORDER BY 需要 filesort 但数据量小可接受。

**预估数据量**：日 200-500 条 → 年 10-20 万 → 5 年 50-100 万

**分表策略（V2 预留）**：
- 当单表 > 1000 万行时按 `YEAR(review_time)` 分表
- 暂不分表，使用归档脚本定期迁移 1 年前数据到历史表

#### 1.2.3 issue（审查问题表）

| 字段 | 类型 | 长度 | 约束 | 默认 | 说明 |
|------|------|------|------|------|------|
| id | BIGINT | 20 | PK, AUTO_INCREMENT | - | 主键 |
| review_id | BIGINT | 20 | NOT NULL | - | 所属审查 |
| rule_id | BIGINT | 20 | NULL | NULL | 触发规则 |
| rule_name | VARCHAR | 100 | NULL | NULL | 规则名（冗余） |
| category | VARCHAR | 20 | NOT NULL | - | STYLE / DEFECT / SECURITY |
| severity | VARCHAR | 20 | NOT NULL | - | CRITICAL / ERROR / WARNING / SUGGESTION |
| line_number | INT | - | NOT NULL | 0 | 行号 |
| end_line | INT | - | NULL | NULL | 结束行号 |
| col_number | INT | - | NULL | NULL | 列号 |
| description | VARCHAR | 500 | NOT NULL | - | 问题描述 |
| suggestion | TEXT | 64KB | NULL | NULL | 修复建议 |
| code_before | TEXT | 64KB | NULL | NULL | 修改前代码 |
| code_after | TEXT | 64KB | NULL | NULL | 修改后代码 |
| ai_explain | TEXT | 64KB | NULL | NULL | AI 解释 |
| is_deleted | TINYINT | 1 | NOT NULL | 0 | 软删除 |

**索引**：
- `PRIMARY KEY (id)`
- `KEY idx_review_id (review_id)` —— 单次审查的问题列表
- `KEY idx_severity (severity)` —— 按严重级别过滤
- `KEY idx_category (category)` —— 按分类过滤
- `KEY idx_review_severity (review_id, severity)` —— **联合索引**，高频查询

**高频查询**：
```sql
-- 单次审查按严重级别 + 行号排序
SELECT * FROM issue
WHERE review_id = ? AND is_deleted = 0
ORDER BY severity, line_number;
```
使用 `idx_review_id` 后 filesort；V2 改用 `idx_review_severity (review_id, severity)` 覆盖。

**预估数据量**：单次审查 10-30 条 → 年 100-500 万 → **最热表**

**分表策略（V2 预留）**：与 review 同周期，按 `review_id` 范围分片

### 1.3 索引使用规范（强制）

参考概要设计 4.3 + 8.2：
- ✅ 选择性 > 5% 的列建索引
- ❌ **禁止**对 TEXT 字段建索引
- ✅ 联合索引遵循最左前缀（`idx_review_severity` 用于 `WHERE review_id=? AND severity=?`）
- ✅ 单表索引 ≤ 6 个
- ❌ 低频查询不建索引
- ❌ 避免 `SELECT *`，按需字段查询

### 1.4 软删除规范

**统一约定**：
- 所有业务表必须有 `is_deleted TINYINT NOT NULL DEFAULT 0`
- `0` = 正常，`1` = 已删除
- MyBatis-Plus `@TableLogic` 自动转换

**MyBatis-Plus 配置**：
```yaml
mybatis-plus:
  global-config:
    db-config:
      logic-delete-field: isDeleted
      logic-delete-value: 1
      logic-not-delete-value: 0
```

**物理删除**：
- 业务代码**禁止**使用 `deleteById` 物理删除
- 联表 / 归档脚本可物理删除（绕开 MP）
- 定期任务清理 1 年前软删除数据

### 1.5 字段默认值规范

| 字段 | 默认值 |
|------|--------|
| 布尔型 (TINYINT) | 0（false） |
| 时间型 | CURRENT_TIMESTAMP |
| 状态枚举 | 业务默认值（如 'PENDING'） |
| 计数 | 0 |
| 字符串 | NULL（不用空串） |

## 2. Redis Key 设计

### 2.1 命名规范
- 前缀：`code-audit:` 业务前缀，避免与其他系统冲突
- 分隔：冒号 `:` 分层
- 长度：建议 ≤ 64 字符

### 2.2 完整 Key 清单

| Key 模板 | 类型 | 序列化 | TTL | 写入方 | 读取方 | 大小预估 |
|---------|------|--------|-----|--------|--------|---------|
| `code-audit:rules:enabled` | String | JSON | 永久（手动失效） | 后端 | 后端 + 网关 | 10-50KB（13+ 条规则） |
| `jwt:blacklist:{token}` | String | "1" | Token 剩余有效期 | 后端 | 后端 + 网关 | 单 token 几字节 |
| `ratelimit:{userId}:{path}:{minute}` | String | count (数字) | 60s | 网关 | 网关 | 1 数字 |
| `session:{userId}` (V2) | Hash | 多字段 | 30min | 后端 | 后端 + 网关 | 几百字节 |

### 2.3 详细说明

#### 2.3.1 规则缓存
- **Key**：`code-audit:rules:enabled`
- **Value**：启用的规则列表（JSON）
- **TTL**：永久
- **失效**：
  - 规则 CRUD 完成后 `DEL` + `PUBLISH code-audit:rule:updated`
  - 多实例场景下所有实例订阅消息后清本地缓存

**降级**：
```java
// 读取顺序
List<Rule> rules = null;
try {
    rules = (List<Rule>) redisTemplate.opsForValue().get("code-audit:rules:enabled");
} catch (Exception e) {
    log.warn("Redis 不可用，降级到内存缓存");
}
if (rules == null) {
    rules = memoryCache.get();
}
```

#### 2.3.2 JWT 黑名单
- **Key**：`jwt:blacklist:{token}`
- **Value**：`"1"`
- **TTL**：Token 剩余有效期（精确到毫秒）
- **写入**：登出时
- **读取**：每次请求鉴权时

```java
public void blacklist(String token) {
    Claims claims = parse(token);
    long ttl = claims.getExpiration().getTime() - System.currentTimeMillis();
    if (ttl <= 0) return;
    redisTemplate.opsForValue().set("jwt:blacklist:" + token, "1", ttl, TimeUnit.MILLISECONDS);
}
```

#### 2.3.3 限流计数（网关）
- **Key**：`ratelimit:{userId}:{path}:{minute}`，如 `ratelimit:123:/api/review/submit:2026061815`
- **Value**：`INCR` 计数
- **TTL**：60s（窗口期）
- **Lua 脚本**：
```lua
local current = redis.call('INCR', KEYS[1])
if current == 1 then
    redis.call('EXPIRE', KEYS[1], ARGV[1])
end
return current
```
- **判断**：`current > limit` 则 429 Too Many Requests

#### 2.3.4 会话（V2 预留）
- **Key**：`session:{userId}`
- **Type**：Hash
- **Fields**：last_ip, last_login_time, online_devices, ...
- **TTL**：30 分钟滑动

### 2.4 持久化策略

| 模式 | 用途 | 启用 |
|------|------|------|
| RDB | 定期快照，启动恢复 | ✅ 5 分钟一次 |
| AOF | 每次写追加，精确恢复 | ✅ `appendfsync everysec` |
| 混合 | RDB + AOF | ✅ Redis 7.0 默认 |

```yaml
# docker-compose.yml
redis:
  command:
    - "redis-server"
    - "--appendonly"
    - "yes"
    - "--appendfsync"
    - "everysec"
    - "--maxmemory"
    - "256mb"
    - "--maxmemory-policy"
    - "allkeys-lru"
```

### 2.5 内存预估

| 数据 | 大小 | 数量 | 合计 |
|------|------|------|------|
| 规则缓存 | 20KB | 1 | 20KB |
| JWT 黑名单 | 100B | 100 在线 | 10KB |
| 限流计数 | 30B | 1000/min | 30KB |
| 会话 (V2) | 500B | 500 在线 | 250KB |
| **合计** | | | **~300KB** |

256MB 内存远足够。

### 2.6 故障应对

| 故障 | 影响 | 应对 |
|------|------|------|
| Redis 宕机 | 鉴权慢、降级到内存规则、限流失效 | 业务可用 + 网关降级无限流 |
| Redis 内存满 | LRU 淘汰规则缓存 | 业务从 MySQL 重新加载 |
| Redis 网络抖动 | 偶发超时 | 重试 3 次 + 降级 |

## 3. 数据迁移与初始化

### 3.1 初始化流程

```bash
# 1. 启动 MySQL 容器
docker compose up -d mysql

# 2. 等待健康检查通过（约 10s）
docker compose ps mysql

# 3. 自动执行 /docker-entrypoint-initdb.d/init.sql
#    - DROP/CREATE database code_audit
#    - CREATE 6 张表
#    - INSERT 3 个默认用户
#    - INSERT 13 条内置规则
```

### 3.2 字段变更策略

**绝对禁止**：
- ❌ 删字段
- ❌ 改字段名
- ❌ 改字段类型（精度退化）

**允许操作**：
- ✅ 加字段（NULLable 或带 DEFAULT）
- ✅ 加索引
- ✅ 加表

变更必须：
1. 先写迁移 SQL（`migration/V1.0.1__add_field.sql`）
2. 在测试环境验证
3. 备份生产数据
4. 执行迁移
5. 验证

V2 引入 Flyway/Liquibase 自动化迁移。

### 3.3 部署脚本

```bash
# docker-entrypoint-initdb.d/init.sql 由 MySQL 容器首次启动自动执行
# 数据卷挂载：./code-audit-server/src/main/resources/db/init.sql:/docker-entrypoint-initdb.d/init.sql:ro
```

## 4. 备份与恢复

### 4.1 MySQL 备份

#### 4.1.1 全量备份
```bash
# 每天凌晨 2 点执行
docker exec code-audit-mysql mysqldump \
    -uroot -proot123 \
    --single-transaction \
    --routines \
    --triggers \
    --events \
    code_audit | gzip > /backup/code_audit_$(date +%Y%m%d).sql.gz
```

#### 4.1.2 binlog 增量
```yaml
# my.cnf
[mysqld]
log-bin = /var/lib/mysql/mysql-bin
binlog-format = ROW
expire-logs-days = 7
```

#### 4.1.3 恢复
```bash
# 恢复全量
gunzip < code_audit_20260618.sql.gz | mysql -uroot -proot123 code_audit

# 重放 binlog
mysqlbinlog mysql-bin.000123 | mysql -uroot -proot123
```

### 4.2 Redis 备份

#### 4.2.1 RDB 快照
- 自动：每 5 分钟（如有 1 个 key 变更）
- 手动：`BGSAVE` 或 `SAVE`

#### 4.2.2 AOF 重写
- 定期：`BGREWRITEAOF`
- 文件：/data/appendonly.aof

#### 4.2.3 恢复
- 启动 Redis 时自动加载 RDB + AOF
- 持久化数据**不关键**（可从 MySQL 重建）

### 4.3 备份保留策略

| 数据 | 保留 | 存储 |
|------|------|------|
| MySQL 全量 | 7 天 | 对象存储 OSS |
| MySQL binlog | 7 天 | 本地 + 异地 |
| Redis RDB | 3 天 | 本地 |
| Redis AOF | 3 天 | 本地 |

## 附录 B：常用查询 SQL 模板

### B.1 审查详情（个人）
```sql
SELECT r.*, u.username
FROM review r
JOIN user u ON r.user_id = u.id
WHERE r.id = ? AND r.is_deleted = 0;
```

### B.2 审查问题（按严重级别排序）
```sql
SELECT *
FROM issue
WHERE review_id = ? AND is_deleted = 0
ORDER BY
    CASE severity
        WHEN 'CRITICAL' THEN 0
        WHEN 'ERROR' THEN 1
        WHEN 'WARNING' THEN 2
        WHEN 'SUGGESTION' THEN 3
        ELSE 4
    END,
    line_number;
```

### B.3 高频问题 Top 10
```sql
SELECT category, rule_name, COUNT(*) AS cnt
FROM issue
WHERE is_deleted = 0
GROUP BY category, rule_name
ORDER BY cnt DESC
LIMIT 10;
```

### B.4 7 天趋势
```sql
SELECT DATE(review_time) AS day, COUNT(*) AS cnt
FROM review
WHERE review_time >= DATE_SUB(NOW(), INTERVAL 7 DAY) AND is_deleted = 0
GROUP BY DATE(review_time)
ORDER BY day;
```

### B.5 班级成员
```sql
SELECT u.*
FROM user u
JOIN class_user cu ON u.id = cu.student_id
WHERE cu.class_id = ? AND cu.is_deleted = 0 AND u.is_deleted = 0;
```
