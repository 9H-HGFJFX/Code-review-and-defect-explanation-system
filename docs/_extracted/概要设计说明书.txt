1. 引言
1.1 编写目的
本文档在需求分析说明书的基础上，对代码审查与缺陷解释系统开展整体概要设计，明确系统整体架构、技术选型、模块划分逻辑、数据表结构、接口通信规范与全域安全策略。屏蔽底层技术实现细节，为后续详细设计、后端编码开发、前端页面开发、联调测试提供统一、可落地的技术框架与开发标准，保证前后端开发口径一致。
1.2 设计原则
分层架构：表现层、网关层、业务层、数据层四层严格分离，层级单向依赖，杜绝跨层调用，最大限度降低代码耦合度，方便后续迭代维护。
前后端分离：前端采用Vue3独立单页应用，后端提供标准Spring Boot RESTful接口，前后端通过JSON数据交互，各司其职、并行开发。
规则可配置热更新：全部代码审查规则统一持久化至数据库，支持后台可视化编辑、启停、修改；规则变更后无需重启服务，实现内存规则缓存秒级热更新。
安全优先：安全扫描引擎独立运行，不与常规代码规范检测耦合；最终审查报告中，安全类缺陷优先级最高，优先置顶展示，提前规避线上高危漏洞。
规模可控：设置统一代码提交行数阈值，单次最大提交代码限制为5000行，前端输入框实时行数校验+后端接口二次校验双重拦截，超限直接拒绝审查请求，避免大代码量导致服务卡顿。
1.3 系统运行约束（新增，补齐边界说明）
语言约束：系统仅支持Java语言代码静态审查，不支持Python、C++、Go等其他编程语言。
环境约束：后端服务运行依赖JDK17，前端适配Chrome、Edge、Firefox主流现代浏览器，不兼容IE浏览器。
业务约束：区分学生、教师、管理员三级权限，教师可管理班级与自定义审查规则，学生仅可提交代码、查看个人审查记录，管理员拥有全系统管理权限。
2. 系统架构设计
2.1 整体架构
系统采用B/S浏览器服务端架构，全程前后端分离，整体分为四层架构，同时补充完整请求调用链路，明确各层交互逻辑：
完整请求链路：前端页面请求→Nginx反向代理→Spring Cloud Gateway网关（鉴权、限流、路由）→后端Spring Boot业务服务→Redis缓存/MySQL数据库。
2.2 技术选型
2.3 部署架构
系统支持传统服务器部署、Docker容器化部署两种模式，统一端口规划：Nginx:80/443、后端服务:8080、MySQL:3306、Redis:6379；当前版本仅做单机部署，集群高可用纳入后续迭代规划。
模式一：传统部署
Nginx 作为反向代理和静态资源服务器，统一转发前后端请求
Spring Boot JAR包独立运行，无服务拆分
MySQL + Redis独立安装部署，本地持久化数据
模式二：Docker容器化部署
前端容器：Nginx + Vue3编译后静态资源
后端容器：Spring Boot业务应用
数据库容器：MySQL 8.0（配置数据卷挂载，防止容器销毁数据丢失）
缓存容器：Redis 7.0（配置数据持久化）
统一使用Docker Compose完成多容器一键编排、启停与管理
3. 模块设计
3.1 模块结构图
系统原有6大核心模块，补充新增班级与成员管理模块，共计7大核心模块，补齐ER图对应的班级业务，模块间全部通过RESTful API异步通信，模块依赖关系：权限相关模块依赖认证模块，审查引擎为公共基础模块，全用户通用。
3.2 审查引擎详细设计
3.2.1 引擎工作流程
补充超时熔断、异常捕获、同步异步区分机制，规避接口卡死问题，完整流程如下：
代码接收：接收前端提交的代码文本与基础文件信息
预处理：代码行数双重校验、编码格式检测、移除文件BOM头；单次审查最大超时时间30s，超时自动终止任务
AST解析：使用JavaParser将代码解析为抽象语法树；代码语法错误直接终止流程，返回语法异常提示
规范检查：遍历AST节点，应用代码规范规则集
缺陷检测：对AST进行数据流和控制流分析，匹配代码缺陷模式
安全扫描：对AST和源码文本进行安全模式匹配；执行优先级：安全扫描>缺陷检测>规范检查
建议生成：根据规则模板和问题上下文自动生成可直接复用的修复建议
报告组装：汇总所有问题，按严重级别从高到低排序，生成最终审查报告
同步异步策略：500行以内代码同步实时返回结果；4000-5000行超大代码异步后台处理，前端轮询查询审查结果。
3.2.2 规则引擎设计
采用“规则定义 + 规则执行器”的设计模式，细化热更新执行逻辑：
规则定义：存储在数据库rule表中，包含规则ID、类别、名称、严重级别、匹配模式、建议模板，区分全局公共规则、班级私有规则
规则执行器：各类规则对应独立的执行器类，统一实现RuleExecutor接口，方便单独启停某一类检测规则
规则加载：系统启动时从数据库加载所有启用规则至本地内存缓存，提升审查响应速度
热更新流程：规则编辑/启停→数据库更新→Redis发布订阅消息→所有服务节点清空本地缓存→自动重新加载最新规则，全程无需重启服务
3.2.3 规范检查引擎
规范检查引擎基于AST节点遍历实现，对每个编译单元（CompilationUnit）进行递归分析：
命名检查器（NamingChecker）：检查类名、方法名、变量名、常量名的命名风格是否符合Java开发规范
格式检查器（FormatChecker）：检查代码缩进、单行行宽、大括号书写风格等格式规范
注释检查器（CommentChecker）：检查类/方法Javadoc完整性、行内注释覆盖率
结构检查器（StructureChecker）：检查方法长度、类文件长度、方法参数个数等代码结构指标
3.2.4 缺陷检测引擎
缺陷检测引擎综合使用AST分析和数据流分析技术：
空指针检测：追踪变量完整赋值路径，精准识别未做非空判断的危险方法调用
资源泄漏检测：分析try-with-resources语法使用情况，识别文件流、网络流等未关闭资源
逻辑错误检测：对条件表达式进行常量折叠，识别代码中永真、永假无效条件
死循环检测：分析循环变量增减修改路径，识别无出口、存在卡死风险的死循环代码
3.2.5 安全扫描引擎
安全扫描引擎采用模式匹配和污点分析相结合的方式，优先拦截高危安全漏洞：
SQL注入检测：追踪前端用户可控输入流向，检测直接拼接SQL造成的注入风险
XSS检测：识别前端输出未做转义的危险代码片段
硬编码检测：正则匹配代码中硬编码密码、密钥、令牌等敏感信息
路径遍历检测：分析文件读写路径，识别用户输入可控造成的目录遍历漏洞
4. 数据库设计
4.1 数据库ER图
系统核心实体关系补充中间表说明，完善多对多关联逻辑：
用户（User）1:N 审查记录（Review）—— 一个用户可提交多次代码审查
审查记录（Review）1:N 审查问题（Issue）—— 一次审查可检出多条代码缺陷
审查规则（Rule）1:N 审查问题（Issue）—— 一条缺陷对应一条命中规则
教师（User）1:N 课程班级（Class）—— 一名教师可管理多个班级
课程班级（Class）N:M 学生（User）—— 多对多关系，通过class_user班级成员中间表完成关联
4.2 核心表结构
全文统一补齐全部枚举取值、补充逻辑删除字段、补充字段默认值，解决原有枚举空白、数据无法软删除问题
4.2.1 用户表（user）
4.2.2 审查记录表（review）
4.2.3 审查问题表（issue）
4.2.4 审查规则表（rule）
4.2.5 新增数据表（补齐业务缺失）
班级表（class）：存储班级基础信息，绑定管理教师ID
班级成员中间表（class_user）：解决班级与学生多对多关联关系，隔离班级数据权限
4.3 索引设计
新增高频业务联合索引，优化查询性能，同时补充索引使用规范：禁止对大文本TEXT字段建立索引，避免降低数据库写入性能。
5. 接口设计
5.1 接口规范
所有API遵循RESTful设计风格，补充统一分页响应格式、业务错误码体系，完善接口返回规范：
基础响应格式
成功响应：{"code": 200, "message": "success", "data": {...}}
失败响应：{"code": 4xx/5xx/自定义业务码, "message": "错误描述", "data": null}
分页列表统一响应格式
自定义业务错误码
10001：账号或密码错误
10002：Token已过期，请重新登录
20001：代码行数超出限制，拒绝审查
20002：Java代码语法错误，无法解析
5.2 核心接口列表
补充班级全套接口，补齐业务缺口；明确报告导出接口支持PDF/Word双格式，通过请求参数区分。
5.3 接口超时与重试策略
接口超时配置：普通查询接口超时10s，代码同步审查接口超时30s，与引擎熔断超时时间保持一致；导出报告大文件接口超时60s
重试机制：仅对网络抖动类GET查询接口开启重试，最大重试次数2次，重试间隔1s；代码提交、数据新增修改类写接口禁止重试，避免重复提交脏数据
异步任务兜底：异步审查任务后台定时巡检，超时未完成任务自动标记为失败并释放线程资源，防止线程池耗尽
5.4 跨域处理方案
统一在网关层完成全局跨域配置，无需后端各个服务单独配置CORS，放行前端Vue3项目域名、全部请求头与请求方式，同时禁止浏览器携带Cookie跨域，规避跨域安全风险。
6. 安全设计
6.1 权限安全设计（三级RBAC权限模型）
系统基于RBAC实现角色权限隔离，严格划分三类用户权限边界，实现数据权限+功能权限双重管控：
学生角色（STUDENT）：仅可提交代码审查、查看个人记录、下载个人报告，无法查看班级其他同学数据、无法编辑任何审查规则
教师角色（TEACHER）：拥有学生全部权限，可管理名下班级、增减学生成员、配置班级私有审查规则、查看班级全体学生审查数据统计
管理员角色（ADMIN）：系统最高权限，管理全量用户、全局公共规则、查看全平台审查统计数据、管理所有班级信息
权限校验时机：网关层前置校验Token合法性，接口层校验用户角色与接口访问权限，数据层自动拼接数据隔离条件，杜绝越权查询他人数据。
6.2 身份认证安全
采用JWT无状态令牌认证，Access令牌有效期2小时，Refresh令牌有效期7天，支持无感刷新登录状态
用户退出登录时，将当前Access令牌加入Redis黑名单，令牌未过期也无法继续使用
密码全程BCrypt单向加密存储，数据库不保存明文密码，禁止前端传输明文密码接口日志
防暴力破解：同一账号连续5次登录失败，锁定账号15分钟，期间禁止登录
6.3 接口安全防护
接口限流：网关层限流，单用户每分钟最多提交5次代码审查请求，防止恶意刷接口压垮引擎服务
请求参数校验：后端所有接口统一做参数非空、长度、格式校验，防御XSS恶意脚本注入，前端代码输入框做HTML转义
SQL注入防护：依托MyBatis-Plus预编译参数绑定，杜绝原生SQL字符串拼接，同时代码审查引擎内置SQL注入漏洞检测能力
敏感信息脱敏：用户邮箱、密码等敏感字段在接口返回与日志打印时自动脱敏，避免信息泄露
6.4 数据安全设计
所有业务数据表统一增加逻辑删除字段，采用软删除方式，防止误删数据无法恢复
数据库开启定时全量备份，每日凌晨2点自动备份，保留最近7天备份文件
Redis缓存数据设置过期时间，避免缓存数据长期堆积、缓存与数据库数据不一致
班级数据物理隔离，学生无法跨班级查询数据，教师仅可查看自己名下班级数据
6.5 代码引擎自身安全
限制单次代码最大行数与最大超时时间，防止恶意超大代码文件导致引擎OOM内存溢出
禁止解析外部恶意远程代码文件，仅支持前端文本直接输入，不支持远程URL代码拉取
引擎运行线程隔离，单条审查任务独立线程执行，单个任务异常不会影响整体服务运行
7. 异常处理设计
7.1 全局异常分类
系统将异常分为四大类，统一全局异常处理器捕获，前端无需分别处理各类后端报错：
参数异常：请求参数非法、代码行数超限、参数格式错误
权限异常：Token过期、无接口访问权限、越权访问他人数据
业务异常：代码语法解析失败、审查任务超时、规则加载失败
系统未知异常：数据库宕机、缓存服务断开、服务内部未知报错
7.2 异常处理流程
接口抛出异常→全局异常拦截器统一捕获
匹配异常类型，返回对应标准化错误码与前端友好提示文案
后端自动打印完整异常堆栈日志，方便开发人员定位问题
屏蔽后端原生报错堆栈信息，不对外暴露服务底层架构细节
7.3 核心业务异常兜底方案
Redis缓存宕机：自动降级为数据库直查，保障审查规则正常加载，仅牺牲部分接口响应性能
代码解析超时：强制终止任务，返回友好提示，同时记录超时日志用于后续引擎性能优化
数据库短暂断开：接口返回服务繁忙提示，配合前端重试机制，等待数据库自动重连
8. 性能设计
8.1 性能指标要求
8.2 性能优化方案
缓存优化：高频访问的审查规则全量缓存至Redis，减少数据库重复查询；用户登录会话信息缓存至Redis，提升鉴权速度
线程池隔离：单独配置代码审查任务线程池，和普通业务接口线程池物理隔离，大代码量审查任务不会阻塞日常查询接口
分页强制约束：所有列表查询接口强制分页，禁止无条件全表查询，避免大数据量查询拖慢数据库
异步解耦：大代码量审查、报告生成、数据统计计算全部异步化，缩短前端接口阻塞等待时间
9. 部署与运维设计
9.1 环境依赖清单
基础运行环境：JDK17、Maven3.9、Node.js18
中间件：MySQL8.0、Redis7.0、Nginx1.24+
最低服务器配置：2核4G，磁盘空间≥40G
9.2 运维监控要点
服务监控：监控后端服务CPU、内存、线程池状态，重点监控审查引擎线程池堆积数量
数据库监控：监控慢查询SQL、连接池占用数，及时优化低效查询语句
日志运维：区分业务日志、异常日志、引擎运行日志，日志按天切割，保留30天日志记录
9.3 回滚方案
版本迭代更新出现故障时，支持一键回滚：容器化部署直接回退至上个镜像版本；传统Jar包部署替换旧包重启服务，数据库变更脚本提前编写反向回滚语句，保障线上故障快速恢复。
10. 需求追溯与未实现功能说明
10.1 需求追溯
