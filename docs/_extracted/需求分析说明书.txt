1. 引言
1.1 编写目的
本文档旨在对“代码审查与缺陷解释系统”进行全面的需求分析，明确系统的功能需求、非功能需求、用户角色及业务流程。本文档是后续概要设计、详细设计、编码实现和测试验收的基础依据。
预期读者包括：项目开发团队、教学管理人员、系统测试人员及项目验收人员。
1.2 项目背景
在计算机编程教学过程中，学生提交的代码往往存在规范性不足、逻辑缺陷、安全隐患等问题。传统的代码审查依赖教师逐行人工审查，效率低下且反馈周期长。本系统旨在构建一个自动化的代码审查平台，对教学场景中的代码进行规范检查、缺陷识别和修改建议生成，辅助教师提升教学效率，帮助学生快速定位和改进代码问题。

2. 系统概述
2.1 系统定位
本系统定位于教学辅助工具，面向高校编程课程的教学场景，提供轻量级、即时反馈的代码审查服务。系统限定代码规模（单文件不超过500行），聚焦于常见教学语言（Java），输出结构化的审查报告。
2.2 用户角色
2.3 系统边界
系统聚焦于教学代码的静态审查，不包括以下能力：
代码编译与运行（仅做静态分析）
复杂项目的架构级审查
运行时性能分析
第三方代码仓库的深度集成（如GitHub全量分析）
3. 功能需求
3.1 功能模块总览
3.2 代码提交与展示（M01）
3.2.1 功能描述
用户可通过文本粘贴或文件上传两种方式提交Java源代码。系统对代码进行语法高亮渲染展示，并限制单次提交代码不超过500行。
3.2.2 输入/输出
3.3 规范检查引擎（M02）
3.3.1 功能描述
基于预定义的编码规范规则库，对提交的代码进行静态检查。检查项覆盖命名规范、代码格式、注释规范、代码结构等方面。
3.3.2 检查项列表
3.4 缺陷检测引擎（M03）
3.4.1 功能描述
通过AST分析和模式匹配，识别代码中的常见缺陷模式。缺陷检测聚焦于教学场景中高频出现的逻辑错误和编码失误。
3.4.2 缺陷类型
3.5 安全扫描引擎（M04）
3.5.1 功能描述
安全扫描引擎对代码进行安全风险检测，重点识别OWASP Top 10中与代码层面相关的安全漏洞。系统在审查报告中以显著方式标注安全问题，并提供安全提示。
3.5.2 安全检测项
3.6 修改建议生成（M05）
3.6.1 功能描述
针对每个检测到的问题（规范、缺陷、安全），系统自动生成修改建议。建议包含：问题描述、修复方案说明、修改前后代码对比。
3.6.2 建议内容结构
问题定位：文件名、行号、问题代码片段
问题说明：简明描述该问题是什么、为什么需要修复
修复方案：文字说明修复策略
代码示例：修改前/修改后的代码对比
3.7 审查报告输出（M06）
3.7.1 功能描述
审查完成后生成结构化报告，包含审查概要、问题清单（按严重级别分类）、详细建议列表。支持在线查看和PDF/Word导出。
3.7.2 报告结构
审查概要：审查时间、代码文件信息、代码行数、问题总数
严重级别统计：严重/错误/警告/建议 各级别问题数量
问题清单：按严重级别排序，每个问题包含位置、描述、建议
安全提示专区：独立板块展示所有安全问题，醒目标注
3.8 规则库管理（M07）
管理员和教师可对审查规则进行管理，包括：查看规则列表、新增自定义规则、编辑规则参数、启用/禁用规则。规则支持按类别（规范/缺陷/安全）分组展示。
3.9 历史记录管理（M08）
系统保存每次审查的完整记录，包括原始代码、审查结果、生成时间。用户可查看历史审查列表、查看详情、对比两次审查结果的差异。
3.10 用户管理（M09）
系统支持用户注册与登录，采用JWT令牌认证。角色分为学生、教师、管理员三级，不同角色拥有不同功能权限。
3.11 数据统计看板（M10）
为教师和管理员提供可视化数据看板，展示：班级代码质量趋势、高频问题Top10、各规范类别违规分布、学生个人代码质量变化曲线。
4. 非功能需求
4.1 性能需求
单次代码审查响应时间 ≤ 5秒（500行代码以内）
系统支持50用户并发访问
页面首次加载时间 ≤ 3秒
审查报告导出（PDF/Word）生成时间 ≤ 2秒
4.2 安全性需求
用户密码采用BCrypt加密存储
API接口采用JWT Token认证
用户提交的代码在服务器端沙箱隔离存储
防止XSS、CSRF、SQL注入等Web安全攻击
敏感操作（规则修改、用户删除）记录审计日志
4.3 可用性需求
系统可用性 ≥ 99.5%（非计划停机 ≤ 3.6小时/月）
界面设计遵循WCAG 2.1 AA级无障碍标准
支持Chrome、Edge、Firefox主流浏览器最新两个版本
4.4 可维护性需求
后端采用分层架构（Controller-Service-Repository）
前端采用组件化开发（Vue3 Composition API）
代码审查规则采用可配置化设计，支持热更新
核心模块单元测试覆盖率 ≥ 70%
4.5 约束条件
开发语言：后端Java 17+，前端Vue3 + TypeScript
框架：Spring Boot 3.x，Node.js 24+
数据库：MySQL 8.0 或 MariaDB 10.6+
部署环境：Linux服务器（Ubuntu 22.04+）或Docker容器
代码规模限制：单次审查不超过500行Java代码
5. 数据需求
5.1 核心数据实体
5.2 数据量估算
用户数：预计500-2000人（中型高校一个院系规模）
日审查次数：预计200-500次
单次审查问题数：平均10-30个
年数据增量：约50万条审查记录 + 500万条问题记录
6. 验收标准
6.1 功能验收
6.2 重点验收项
（1）限定代码规模
系统拒绝审查超过500行的代码文件
提交页面明确提示代码行数限制
上传文件时进行行数预检
（2）安全提示
安全类问题在报告中以红色/橙色醒目标注
安全提示包含风险等级、风险说明、修复建议
报告顶部设置“安全风险摘要”区域
（3）错误建议评测
准备标准错误用例集（包含20个典型错误代码样本）
验证系统对每个错误样本的检测准确率 ≥ 85%
验证修改建议的合理性（人工评估 ≥ 80%建议为有效建议）
记录误报（False Positive）和漏报（False Negative）数量
